## What this project is

`logged-stream` is a small, published Rust **library crate** (crates.io: `logged-stream`,
docs.rs, GitHub: `qwerty541/logged-stream`). It provides a single wrapper type,
[`LoggedStream`](src/stream.rs), that wraps an underlying IO object and logs every
read, write, error, shutdown, and drop that passes through it.

- **Purpose:** transparent logging/inspection of IO traffic (network sniffing, protocol
  debugging, I/O diagnostics, proxy/audit logging).
- **Edition:** Rust 2024. **MSRV:** 1.85.1 (`rust-version` in [Cargo.toml](Cargo.toml)).
- **Runtime deps:** `chrono` (timestamps), `itertools`, `log` (logging facade),
  `tokio` (only `io-util`, for the async traits). Dev-only: `criterion`, `env_logger`,
  more `tokio` features.
- **No `unsafe`.** No `tests/` integration directory ŌĆö tests are inline `#[cfg(test)]`
  modules plus rustdoc doctests.

## Architecture ŌĆö the four-part composition

`LoggedStream<S, Formatter, Filter, L>` is generic over four independent, pluggable
parts. A record flows: **event ŌåÆ Formatter ŌåÆ Filter ŌåÆ Logger**.

1. **`S` ŌĆö the inner IO object.** Sync (`std::io::Read` / `std::io::Write`) or async
   (`tokio::io::AsyncRead` / `AsyncWrite`). `LoggedStream` re-implements the same trait
   it wraps, so it is a drop-in replacement.
2. **`Formatter: BufferFormatter`** ŌĆö turns byte buffers into display strings.
3. **`Filter: RecordFilter`** ŌĆö decides which [`Record`](src/record.rs)s are logged.
4. **`L: Logger`** ŌĆö the sink that consumes accepted records.

### Module map (`src/`)

| File | Responsibility |
| --- | --- |
| [lib.rs](src/lib.rs) | Crate root: `mod` declarations, `pub use` re-exports (flat public API), crate-level rustdoc. |
| [stream.rs](src/stream.rs) | The `LoggedStream` struct and every trait impl: sync `Read`/`Write`, async `AsyncRead`/`AsyncWrite`, `Debug`, and `Drop`. This is where events become `Record`s. |
| [record.rs](src/record.rs) | `Record { kind, message, time }` and `RecordKind` (`Open`, `Read`, `Write`, `Error`, `Shutdown`, `Drop`), each mapped to a display char (`+ < > ! - x`). |
| [buffer_formatter.rs](src/buffer_formatter.rs) | `BufferFormatter` trait + 5 concrete formatters generated by the `define_formatter!` macro. |
| [filter.rs](src/filter.rs) | `RecordFilter` trait + `DefaultFilter`, `RecordKindFilter`, and the composite `AllFilter` (AND) / `AnyFilter` (OR). |
| [logger.rs](src/logger.rs) | `Logger` trait + `ConsoleLogger`, `MemoryStorageLogger`, `ChannelLogger`, `FileLogger`. |

### Provided implementations

- **Formatters:** `LowercaseHexadecimalFormatter`, `UppercaseHexadecimalFormatter`,
  `DecimalFormatter`, `OctalFormatter`, `BinaryFormatter`. All are generated by one
  macro, store a separator as `Cow<'static, str>` (default `:`), and expose parallel
  constructors: `new`, `new_static`, `new_owned`, `new_default`, plus `From`/`Default`/
  `Display`/`Eq`/`Hash`.
- **Filters:** `DefaultFilter` (accepts all), `RecordKindFilter` (allow-list of kinds),
  `AllFilter`/`AnyFilter` (combine `Box<dyn RecordFilter>`; empty `AllFilter` is `true`,
  empty `AnyFilter` is `false`).
- **Loggers:** `ConsoleLogger` (via `log` facade), `MemoryStorageLogger` (bounded
  `VecDeque`), `ChannelLogger` (`mpsc`), `FileLogger` (writes to a file).

## Conventions and invariants to preserve

These are load-bearing ŌĆö breaking them is a breaking API change and must be reflected in
tests, docs, and the changelog.

- **Trait bounds:** `BufferFormatter`, `RecordFilter`, and `Logger` are all
  `Send + 'static` and **object-safe**. Each has a blanket impl for `Box<...>`
  (and `BufferFormatter` also for `Arc<T>`), so trait objects work everywhere.
  The test modules assert `Send` / `Unpin` / object-safety / `Box` for every type ŌĆö
  keep those assertions passing when adding a new implementation.
- **Adding a new formatter/filter/logger:** implement the trait, add a `Box<...>`
  blanket-compatible impl if the pattern calls for it, re-export it from
  [lib.rs](src/lib.rs), and mirror the existing test coverage (`Send`, `Unpin`,
  object-safety, behavior).
- **`RecordFilter::fmt_debug`:** composite filters render children through this hook.
  A new filter with meaningful state should override it (defaults to `"UnknownFilter"`).
- **Import style:** `rustfmt.toml` sets `imports_granularity = "Item"` ŌĆö **one `use`
  per line, no brace-grouped imports** ŌĆö and `use_field_init_shorthand = true`. Match
  the surrounding files exactly; `cargo fmt` enforces this.
- **Docs are duplicated on purpose:** the crate-level overview appears in both
  [lib.rs](src/lib.rs) and on the `LoggedStream` type in [stream.rs](src/stream.rs).
  If you change the list of formatters/filters/loggers or the four-part description,
  update **both** (and the README).

### Behavioral gotchas

- **`RecordKind::Open` is defined but never emitted** by `LoggedStream`. Reads emit
  `Read`, writes `Write`, `poll_shutdown` emits `Shutdown`, `Drop` emits `Drop`
  (`"Deallocated."`), and IO errors emit `Error`. Don't assume `Open` fires.
- **Errors are partly swallowed:** sync `read` ignores `WouldBlock`; sync `write`
  ignores `WriteZero | WouldBlock`; async paths treat `Poll::Pending` / zero-length
  reads as non-events. Only "real" errors produce an `Error` record.
- **Filter runs on every record kind**, including `Drop` and `Shutdown` ŌĆö a
  `RecordKindFilter` that omits those kinds will suppress their log lines.

## Common commands

```bash
# Build (all targets: lib, examples, benches)
cargo build --all-targets

# Tests ŌĆö CI runs this (lib unit tests + example/bench compile)
cargo test --lib --examples --benches
# Doctests are NOT covered by the line above (a cargo quirk) ŌĆö run them too:
cargo test --doc

# Lint (must be clean; CI denies all warnings)
cargo clippy --all-features -- -D warnings

# Format
cargo fmt --all            # apply
cargo fmt --check          # verify (CI uses this)

# Docs (as built for GitHub Pages, on nightly)
cargo doc --no-deps --all-features

# Examples
cargo run --example tcp-stream-console-logger
cargo run --example tokio-tcp-stream-console-logger
cargo run --example file-logger
cargo run --example composite-filters
RUST_LOG=debug cargo run --example tcp-stream-console-logger   # control verbosity

# Benchmarks (Criterion; harness = false)
cargo bench
cargo bench --bench filter
```

CI ([.github/workflows/check.yml](.github/workflows/check.yml)) runs clippy, fmt, and
build+test across `{ubuntu, macos, windows} ├Ś {stable, beta, nightly}`, plus an MSRV
check (`cargo msrv find`). Keep changes green on **stable** at minimum; avoid
nightly-only features. Avoid raising the MSRV without discussion ŌĆö if it must change,
update `rust-version` in [Cargo.toml](Cargo.toml), the badge in [README.md](README.md),
and note it in the changelog.

## Definition of Done

A change is **not complete** until all of the following hold. Apply judgement about
which items are relevant, but do not skip an applicable one silently ŌĆö call it out.

1. **Tests updated/added.** Any new behavior that is exercised, and any regression that
   is fixed, must be covered by a test (inline `#[cfg(test)]` unit test, or a rustdoc
   doctest for public-API examples). Prefer extending the existing per-module test
   modules and their `Send`/`Unpin`/object-safety assertions. Add a regression test that
   fails before the fix and passes after.
2. **Docs kept in sync.** Update every affected document:
   - **CLAUDE.md** (this file) when architecture, invariants, commands, or workflow change.
   - **[README.md](README.md)** when public API, usage, examples, or supported versions change.
   - **[CONTRIBUTING.md](CONTRIBUTING.md)** when the dev workflow, tooling, or commands change.
   - Rustdoc comments on the touched public items (and the mirrored crate-level docs in
     [lib.rs](src/lib.rs) / [stream.rs](src/stream.rs)).
3. **Changelog updated for user-facing changes.** If the change affects users of the
   crate (public API, behavior, MSRV, dependencies, notable fixes), add an entry to
   [CHANGELOG.md](CHANGELOG.md) under an `## Unreleased` section (create it above the
   latest release if absent), using the existing category headings: *Added, Changed,
   Fixed, Removed, Documentation, Dependencies, Testing*. Follow
   [Keep a Changelog](https://keepachangelog.com/) + SemVer. Internal-only refactors,
   test-only, or CI-only changes do not require a changelog entry.
4. **Full check suite passes locally.** Run and confirm all of these succeed:
   ```bash
   cargo build --all-targets
   cargo test --lib --examples --benches
   cargo test --doc
   cargo clippy --all-features -- -D warnings
   cargo fmt --check
   ```
   Fix issues rather than suppressing them; avoid broad `#[allow(...)]` attributes
   without a strong, stated reason.

Report the outcome honestly: if a check fails or a step was skipped, say so with the
relevant output rather than claiming success.
